# mono-ledger-sync v0.2.0

This is the first major post-launch release: 228 commits since `v0.1.1` covering
a full local-first personal-finance workspace, live-by-default sign-in, a
reporting suite, a recurring-payments engine, a diagnostics endpoint, and a
privacy/security hardening pass. The package crosses the `0.2.x` minor boundary
because it adds user-visible capability (reports, bulk edit, category version
history, diagnostics) rather than only fixing bugs.

## Headline

`mono-ledger-sync` now ships as a complete local-first Monobank personal-finance
workspace. A new user can install, paste a personal API token from
`https://api.monobank.ua/`, and see their real accounts, jars, statements, and
currency rates alongside category budgets, recurring-payment tracking, monthly
reports, and a support bundle for diagnostics — all without the package ever
sending data to a hosted service. Fixture data is no longer reachable from the
public user flow; it remains an explicit development opt-in.

## What's new for users

### Live-by-default sign-in flow

- First-run Vite UI greeting leads with **Sign in with Monobank**, with a
  one-click link to `https://api.monobank.ua/` for first-time token retrieval
  and a masked account summary from `GET /personal/client-info` after a
  successful save. (#338, #342)
- Every route that needs a token — Overview, Transactions, Accounts, Rules &
  Mappings, Sync & Webhooks, Exports — now shows a sign-in prompt with a
  one-click jump to the Settings route instead of rendering fixture demo data.
  (#340)
- A **Re-check Monobank connection** button in workspace settings re-runs
  `GET /personal/client-info` against the live API, refreshes the masked
  account inventory, and surfaces the upstream `errorDescription` on failure.
  (#342)
- New public API endpoint `POST /api/app/token/recheck` returns the same
  `LocalApiMonobankTokenStatus` shape used by `POST /api/app/token`, including
  a structured 400 response with `error`, `message`, and `upstreamStatus` on
  upstream failure. (#342)
- Saving a valid token auto-promotes the active profile to the `monobank`
  source; deleting a token auto-demotes it. `last-synced` and
  `next-sync-allowed` timestamps are surfaced through the local config
  endpoint. (#198, #330, #332)

### Workspace and UI

- Bulk edit controls on the Transactions route (multi-select + apply category,
  tag, and review flag in one action). (#290, #418)
- New **Overview finance summary** card with month-to-date cashflow, account
  and jar sync health, spending by category, upcoming recurring payments, and
  budget progress warnings. (#276, #278, #280, #282, #416)
- **Standardized income/expense colors** at the semantic-token level so every
  screen routes positive and negative money through the same palette. (#420)
- **Category version history**: every category edit is recorded with an
  effective-dated version, so retroactive reporting and audits can reconstruct
  a category's name, color, and parent at any past point. (#422)
- **Local setup help route** for first-run troubleshooting, with a guided
  checklist (token, sync, accounts visible) and links to the diagnostics
  endpoint. (#412)
- **shadcn primitive coverage** is now enforced by tests: every web control
  must compose from the shadcn registry, with semantic-token color routing,
  product-specific UI copy, and `lucide-react` icons only. (#408 and follow-up
  PRs)
- Privacy onboarding screen, offline snapshot browsing, and import/export of
  local configuration. (#268, #270, #272)

### Reporting suite

A full reporting suite runs on the user's local SQLite:

- Monthly spending report
- Cashflow report
- Category trend report
- Merchant trend report
- Savings rate report
- Balance projection report
- All totals are currency-aware. (#372)

### Recurring-payments engine

- Recurring transaction pattern detection
- Recurring payment calendar
- Missed-payment detection
- Subscription increase alerts
- Confirm / ignore controls so false positives don't pollute the calendar

### Diagnostics

- New `GET /api/app/diagnostics` returns a redacted support bundle
  (environment, recent activity, redaction-aware log excerpt) safe to share
  with maintainers. The static redaction checker
  (`scripts/check-unsafe-logging.mjs`) is wired into CI to keep the bundle
  honest. (#426, #430)

## What's new for developers

- **Privacy regression test suite** locks in that token, identifier, and raw
  payload redaction holds across the live and fixture paths. (#445, #344)
- A local **mock Monobank server** drives offline integration tests for sync
  retry (429 handling), the saved-token monobank sync path, webhook delivery
  flows, and the live `/bank/currency` smoke. (#110, #158, #170, #177, #216,
  #336)
- New `LocalApiMonobankClientInfoSummary` type on the web side so the masked
  `clientInfo` field returned by the server is typeable on the web. (#338)
- SQLite **query performance indexes** for the read paths that drive the
  workspace routes. (#234)
- Domain error taxonomy and structured logging redaction are now first-class;
  sync interrupt handling is partial-safe. (#95, #97, #99)

## Privacy, security, and operations

This release also ships a long-running privacy/security hardening pass:

- **Threat model** for the five Milestone 14 threat categories, including
  token handling, web exposure, webhook semantics, identifier leakage, and
  fixture contamination. (#432)
- **Security advisory process** with supported versions, embargo window, and
  disclosure timeline. (#449)
- **Database directory encryption guide** with step-by-step instructions for
  the supported platforms. (#447)
- **Gitleaks secret scan** in CI to catch accidentally committed credentials
  before they reach `main`. (#140)
- **Dependency audit** and **static check for unsafe logging patterns** run on
  every push. (#134, #430)
- **Dependabot** config for npm and GitHub Actions with grouped minor/patch
  updates. (#439)
- **npm trusted publishing** with OIDC for the release workflow, and a
  `npm publish --dry-run` step that validates the manifest and tarball before
  publishing. (#11, #435)
- **Packed-tarball smoke workflow** confirms a real `npm pack` resolves and
  typechecks end-to-end, catching export-path mistakes that the build itself
  would not surface. (#437)
- **Loopback binding** for the local API is now enforced by a CI test so the
  local server can never accidentally bind a public interface without
  passcode protection. (#378)

## Fixed

- **Keep synced Monobank data visible** (#457): the local statement pull
  paginates correctly, the public token is genuinely optional (it never
  overrides a saved personal token), and the first-run flow no longer hides
  already-synced data behind a sign-in prompt.
- The local API binds to `127.0.0.1` by default; binding to a non-loopback
  host requires `MONO_LEDGER_SYNC_ACCESS_PASSCODE`. (#378)
- Sync interruptions are partial-safe and resumable; failed webhook events
  retry, pending webhook windows reconcile, and webhook payload hashes are
  canonicalized so dedup is deterministic. (#99, #244, #250, #252, #254, #256,
  #258, #260, #164)
- Sync refuses to start without a saved token; statement items with missing
  ids fall back to a deterministic identity; the `/personal/webhook` endpoint
  rejects malformed payloads, validates GET requests, and randomizes its
  receiver path. (#105, #112, #114, #120, #122, #124, #128, #132)
- Split-plan drafts validate the total in the UI before save. (#93)
- Structured log redaction is centralized and hardened. (#97, #194)

## Upgrade notes

This is a minor version bump from `0.1.1` to `0.2.0`. No manual action is
required. No package export path has changed. A user upgrading from `0.1.x`
will see:

- A sign-in greeting as the lead element of the Settings route.
- A sign-in prompt on Overview, Transactions, Accounts, Rules & Mappings,
  Sync & Webhooks, and Exports if no token is saved for the active profile.
  Saving a token restores the normal data view.
- The new Overview finance summary card, the reporting suite, the recurring
  payment calendar, and the diagnostics endpoint.
- Standardized income/expense colors and shadcn-only UI controls in every
  screen.

The local API still binds to `127.0.0.1` by default; users who want to expose
it on a non-loopback host must set `MONO_LEDGER_SYNC_ACCESS_PASSCODE`.
