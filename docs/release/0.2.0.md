# mono-ledger-sync v0.2.0 — Live by default

This release promotes the live-by-default posture to the public user flow. A new user with no local workspace can run `npm install`, start the local server, paste a personal Monobank API token from `https://api.monobank.ua/`, and see their real accounts, jars, statements, and currency rates inside the Vite UI without the app ever reading from a sanitized fixture. Fixture data remains available as a developer-only opt-in behind the `fixture` source setting and is no longer reachable from the public user flow.

## Headline

`mono-ledger-sync` now talks to the real Monobank personal API (`https://api.monobank.ua`) by default. The product never silently falls back to fixture data when the token is missing, invalid, or revoked; it surfaces a clear "Sign in with Monobank" prompt instead.

## Added

- A first-run Vite UI greeting that leads with "Sign in with Monobank" instead of "Create workspace", with a one-click link to `https://api.monobank.ua/` for first-time token retrieval and a masked account summary from `GET /personal/client-info` after a successful save. (PR #338, #342)
- A first-run empty state for every route that needs a token. Overview, Transactions, Accounts, Rules & Mappings, Sync & Webhooks, and Exports now show a sign-in prompt with a one-click link to the developer portal and a one-click jump to the Settings route instead of rendering fixture demo data. (PR #340)
- A "Re-check Monobank connection" button in the workspace settings that re-runs `GET /personal/client-info` against the live API, refreshes the masked account inventory, and surfaces the upstream `errorDescription` on failure. (PR #342)
- A live `/bank/currency` smoke test that hits the public Monobank endpoint without a token, gated on the `MONO_LEDGER_SYNC_LIVE_MONOBANK_TESTS` env var and skipped when the network is unavailable. (PR #336)
- A new privacy test suite (`test/live-monobank-privacy.test.js`) that proves the redacted log path does not print the live token, the live `X-Token` header, or the live response payload even when the real API returns a partial error. (PR #344)
- New public API endpoint `POST /api/app/token/recheck` that re-runs the live client-info probe against the saved token and returns the same `LocalApiMonobankTokenStatus` shape used by `POST /api/app/token`, including a structured 400 response with `error`, `message`, and `upstreamStatus` on upstream failure. (PR #342)
- New `LocalApiMonobankClientInfoSummary` type on the web side so the masked `clientInfo` field returned by the server is typeable on the web. (PR #338)

## Changed

- The Settings route now leads with the "Sign in with Monobank" greeting card; the "Create workspace" call to action is now a secondary card below the greeting. (PR #338)
- The fixture source is no longer reachable from the public user flow. It remains a developer opt-in for screenshots, offline development, and contributor onboarding. Production user flows that lack a token now see the sign-in prompt instead. (PR #340)
- The `recheckMonobankConnection()` web client now returns a `RecheckMonobankConnectionResult` union so the destructive-alert error path can read the upstream error message directly. (PR #342)
- The `recheckMonobankToken()` server helper is plumbed through `registerLocalApiRoutes` so the new `POST /api/app/token/recheck` route can call it. (PR #342)

## Fixed

- No public bug fixes in this release.

## Security and privacy notes

- The Monobank personal API token is stored in OS secure storage by default. The local API never relays the token off the device and never sends financial data to a hosted service.
- The redaction layer covers `X-Token`, `X-Key-Id`, `X-Sign`, `X-Request-Id`, the token value itself, IBAN, PAN, counterparty identifiers, and the raw response payload. The new `test/live-monobank-privacy.test.js` suite locks that behavior in.
- The fixture source is explicitly opt-in. A user who never touches the developer settings will never see fixture accounts, transactions, or jars.
- The personal API is for the user's own data only. The README and `SECURITY.md` continue to warn against using personal API tokens for other people's data or hosted business flows.

## Upgrade notes

This is a minor version bump from `0.1.1` to `0.2.0`. No database migration is required. No package export path has changed. A user upgrading from `0.1.x` will see the new sign-in greeting as the lead element of the Settings route and a sign-in prompt on Overview/Transactions/Accounts/Rules/Sync/Exports if no token is saved for the active profile. Saving a token (or switching the source to `fixture` in developer settings) restores the normal data view.
