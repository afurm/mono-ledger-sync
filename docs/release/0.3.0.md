# mono-ledger-sync v0.3.0

The headline of this release is an internal refactor: the local API server and
the Vite web app are split from monolithic entrypoints into per-feature modules
so future work can land in one place at a time. The package crosses into the
`0.3.x` minor because the refactor surfaces a small set of new user-visible
behaviour — manual recurring items, on-the-fly category-rule creation, the
local cockpit workflow settings, the `local_exports` audit log, and richer
ledger export filters — alongside the architecture cleanup.

This release was prepared as a single squash commit against `main` (zero
intermediate commits between `v0.2.1` and `HEAD`); the pre-publish gate
(typecheck, build, tests, browser smoke, format, `npm publish --dry-run`) is
green on the release commit. The release workflow will run on tag push.

## Headline

`mono-ledger-sync` is now a modular local-first Monobank personal-finance
workspace. The local API server (`src/server/index.ts`) is down from 4419 to
1765 lines, with eleven route modules under `src/server/routes/`. The Vite web
app (`src/web/App.tsx`) is down from 11154 to 1111 lines, with ten per-route
components under `src/web/routes/` and a new `src/web/api-types.ts` that owns
the wire-format type surface. The user-facing routes, data shape, and HTTP
endpoints are unchanged unless called out below.

## What's new for users

### Manual recurring items

- New endpoint `POST /api/ledger/recurring-items` accepts a
  `ManualRecurringItemInput` (account, optional category, optional merchant
  name, frequency, optional expected-amount range, active flag, start date)
  and returns the persisted `RecurringItem`. Backed by the new
  `LedgerWriteService.createRecurringItem` write method.
- Manual items flow into the existing upcoming-payments, missed-payments, and
  subscription-increase reports alongside detected candidates.

### On-the-fly category rules

- New endpoint `POST /api/ledger/category-rules` accepts a `CategoryRuleInput`
  and returns the persisted rule. The Transactions route now exposes a
  "create rule from this match" affordance that calls this endpoint, so a
  category assignment can promote itself to a reusable rule in one step.
- Backed by the new `LedgerWriteService.createCategoryRule` write method, with
  deterministic `categoryRuleIdFromInput` / `defaultCategoryRuleName` /
  `nextUserCategoryRulePriority` helpers so two calls with the same logical
  input collapse to the same row.

### Local cockpit workflow settings

Migration `0023_local_app_cockpit_settings` extends `local_app_settings` with
six new columns and `LocalAppSettings` / `LocalAppSettingsUpdate` mirror them
on the domain surface:

- `syncSchedule` — `"manual" | "hourly" | "daily" | "app_start"` controls when
  the local sync runs.
- `excludedAccountIds` — accounts skipped from sync and reports.
- `exportDirectory` — default destination for local exports.
- `budgetWarningThreshold` — percentage that flips a budget into its warning
  state.
- `lastBackupAt` and `lastCompactAt` — timestamps surfaced in the diagnostics
  bundle so support can see when the workspace was last maintained.

### Local export audit log

- Migration `0024_local_exports` adds a `local_exports` table that records
  recent export runs: id, profile, format, preset, filter set, row count,
  destination, file path, status, created/completed timestamps, and any error
  message. Indexed on `(profile, created_at DESC)`.
- `LedgerExport` now returns the rendered `format`, optional `preset`, the
  filter set that was applied, and the resolved `rowCount` so the audit row
  is the exact request that produced the file on disk.

### Ledger export filters

`ExportRequest` gains six filter dimensions: `merchantName`, `status`
(`"hold" | "posted"`), `reviewState` (one of the new ledger review states),
`currencyCode`, `amountMin` / `amountMax`, and `includeExcludedAccounts`. The
underlying `LedgerEntryQuery` carries the same dimensions, so the same filter
set works for both the export route and the transactions list.

### Ledger review states

- Migration `0022_ledger_entry_review_states` adds a per-profile review-state
  table keyed by `ledger_entry_id` with a default of `"needs_review"` and
  indexed on `(profile, state, updated_at DESC)`.
- `updateLedgerEntriesBulkEdit` accepts `reviewState` and `reviewedSource`;
  the bulk-edit UI on the Transactions route can now flip a batch into
  `"reviewed"` and surface a `reviewedAt` timestamp.
- A `LedgerEntry.reviewState` field is propagated through the storage and web
  type layers so the UI can render badges and filter by review state.

### Reports

- `isTransferLikeEntry` now excludes internal transfers from the cashflow and
  savings-rate report aggregations, so the headline cashflow figure is the
  user's actual income minus spending rather than a gross transfer-inclusive
  number.

## What's new for developers

### Server modularization

- `src/server/index.ts` is reduced from 4419 to 1765 lines. The remaining
  content is the Fastify bootstrap, webhook path generation, passcode
  resolution, the diagnostics-token probe, and the route-registration fan-out
  — i.e. the things that genuinely are cross-cutting. Everything per-feature
  is delegated to a route module.
- Eleven new files under `src/server/routes/` split the HTTP surface by
  concern, with a single `shared.ts` holding the common route-context type
  and the input-decoding helpers (`readStringQuery`, `readNumberQuery`,
  `readBooleanQuery`, `readUtcDateQuery`, `readManualRecurringItemInput`,
  `readCategoryRuleInput`):
  - `app-routes.ts` — health, token, profile, settings, diagnostics-token
    probe, sign-in recheck, fixtures dev mode.
  - `diagnostics-routes.ts` — `GET /api/app/diagnostics`, support bundle
    endpoint.
  - `export-routes.ts` — ledger export, local-configuration export, import
    flow, plus the `local_exports` audit-log rows.
  - `ledger-routes.ts` — accounts, categories, jars, budgets, rules,
    category-rule create, merchants, merchant-cleanup rules, monthly
    budgets, savings goals.
  - `recurring-routes.ts` — detection candidates, confirm/ignore, calendar,
    missed, subscription-increase alerts, manual items.
  - `report-routes.ts` — monthly spending, cashflow, category trend, merchant
    trend, savings rate, balance projection.
  - `sync-routes.ts` — sync run control, sync history, schedule.
  - `transaction-routes.ts` — ledger entries, annotations, bulk edit, splits,
    review state, splits review.
  - `web-routes.ts` — Vite dev / built-asset serving in production.
  - `webhook-routes.ts` — randomized-path Monobank webhook receiver.
- 63 endpoints across the eleven files, with Fastify JSON schemas co-located
  with the handler that owns them.

### Web modularization

- `src/web/App.tsx` is reduced from 11154 to 1111 lines. The remaining file
  is the router shell, layout, and the small handful of cross-route UI
  primitives that the route components compose.
- Ten new files under `src/web/routes/` own one screen each:
  - `overview/index.tsx` (1900 lines) — finance summary, account/jar health,
    category spending, upcoming recurring, budget warnings.
  - `transactions/index.tsx` (3100 lines) — ledger transactions, filtering,
    pagination, bulk edit, review state, splits, annotations, the new
    on-the-fly category-rule creation.
  - `accounts/index.tsx` (543) — connected accounts and the new per-account
    `includedInReports` toggle surfaced on the domain.
  - `exports/index.tsx` (606) — export workflows and the `local_exports`
    audit log.
  - `rules/index.tsx` (2053) — automation rules, category rule edits, MCC
    rules, merchant-cleanup rules.
  - `recurring/index.tsx` (1288) — recurring-payments calendar, missed,
    subscription-increase alerts, the new manual-items surface.
  - `reports/index.tsx` (composed from `report-cards.tsx`) — the reporting
    suite.
  - `settings/index.tsx` (1564) — token, profile, sign-in recheck, fixtures
    dev mode, the new cockpit workflow settings.
  - `sync/index.tsx` (1308) — sync control, schedule, webhooks, recent runs.
  - `help/index.tsx` (275) and `logs/index.tsx` (237) — first-run help and
    the diagnostics timeline.
- New shared modules: `src/web/api-types.ts` (913 lines) owns the wire-format
  type surface that used to live inline in `api.ts`; `src/web/report-cards.tsx`
  (1821 lines) owns the report card components; `src/web/transaction-cells.tsx`
  (579 lines) owns the memoized table-cell primitives used by the
  Transactions route; `src/web/status.ts` (65 lines) owns the
  status-variant / activity-event-type / token-state label helpers.
- The `api.ts` call surface is unchanged; the route components reach it
  through the same `loadX` / `createX` / `updateX` exports as before.

### Type surface

- `LocalAppSettings` / `LocalAppSettingsUpdate` gain the six cockpit
  workflow fields listed above.
- `LedgerAccount` gains an optional `includedInReports: boolean` so the
  Accounts route can toggle an account in or out of the report aggregations.
- `LedgerEntry` propagates `reviewState` so the Transactions list and the
  export filter can read and filter on it.
- `LocalExportRecord` is exported from `src/storage/index.ts` so the export
  route can write audit rows through the same service the rest of the app
  uses.

## CI / release workflow

- The release workflow now installs the Playwright Chromium browser and runs
  `npm run smoke:web` between the unit tests and the format check. `smoke:web`
  builds the Vite dev server, hits every public route with a real headless
  browser, and asserts the rendered HTML contains the expected page heading.
  The PR template adds a corresponding `npm run smoke:web` checkbox so the
  same gate is documented at the PR level. The thirteen routes verified are
  `overview`, `transactions`, `categories`, `budgets`, `recurring`, `reports`,
  `rules`, `sync`, `accounts`, `exports`, `logs`, `settings`, `help`.

  unsafe-logging check, the dependency audit, or Dependabot in this release —
  those landed in `0.2.0` and continue to gate the pipeline.
- `npm trusted publishing` with OIDC and the `npm publish --dry-run` step

## Fixed

- The `App.tsx` → per-route split surfaced and corrected a handful of stale
  imports that had drifted across the old monolith during the `0.2.x`
  hardening pass; the new files compile and smoke-OK on every public route.
- The `api.ts` → `api-types.ts` split closes the loop on a long-standing
  complaint that wire-format types and API call wrappers shared one file
  with no clear boundary; the new layout keeps the call wrappers lean and
  groups the type surface where it can be reached by both the web routes
  and the server routes without an import cycle.

## Upgrade notes

This is a minor version bump from `0.2.1` to `0.3.0`. No manual action is
required. A user upgrading from `0.2.x` will see:

- A new "create rule from this match" affordance on the Transactions route.
- A new "manual recurring item" surface on the Recurring Payments route.
- The `0.2.0` reporting suite now excludes internal transfers from the
  cashflow and savings-rate totals; figures between `0.2.x` and `0.3.0` are
  not directly comparable on those two reports if the user has recorded
  transfers.
- The Settings route exposes the new cockpit workflow settings; existing
  values are read from the migration as `NULL` and treated as "use default"
  by the UI.
- The Exports route now writes a `local_exports` audit row for every run;
  the audit log is the file the support bundle reads when computing
  diagnostics.

The local API still binds to `127.0.0.1` by default; users who want to
expose it on a non-loopback host must set
`MONO_LEDGER_SYNC_ACCESS_PASSCODE`. No package export path has changed;
the public exports of `mono-ledger-sync`, `mono-ledger-sync/core`,
`mono-ledger-sync/exports`, `mono-ledger-sync/server`, and
`mono-ledger-sync/web` are unchanged.
