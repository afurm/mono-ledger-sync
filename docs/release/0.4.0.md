# mono-ledger-sync v0.4.0

The `0.4.0` release is the portability and provider-spike slice. It makes local
ledger data easier to move into accountant and BI workflows without changing the
default product boundary: the app still runs locally, talks to Monobank through
the user's personal token, and keeps provider mode experimental.

## Headline

`mono-ledger-sync` now exports BI-ready files and redacted database snapshots.
Parquet exports, DuckDB-friendly SQLite views, local BI docs, and accountant
handoff docs give power users a practical path from the local ledger to external
analysis tools. The release also validates the first provider/FOP signing and
mock-server boundary, but provider mode does not graduate into the default UI.

## What's New For Users

### BI-ready exports

- `format=parquet` is supported by the local export API and Exports route.
  Parquet files include flat transaction columns, UTC date/month helpers,
  integer minor-unit money fields, major-unit helper columns, review state,
  notes, tags JSON, and split-plan JSON.
- Redacted SQLite snapshots are available through
  `format=sqlite&redacted=true`. They keep normalized ledger rows and BI views
  while removing raw statement rows, webhook payload rows, raw Monobank JSON,
  masked PAN JSON, local export paths, and export-directory settings.
- The Exports route now exposes Parquet and SQLite snapshot formats directly in
  the wizard. SQLite snapshots default to redacted mode.

### Local BI and accountant handoff

- Migration `0026_bi_views` adds five DuckDB-friendly views:
  `v_transactions_long`, `v_monthly_spending`, `v_daily_balance`,
  `v_recurring_commitments`, and `v_budget_progress`.
- New docs cover the local DuckDB workflow, local-only Metabase/Grafana setup,
  and accountant handoff expectations:
  - `docs/integrations/duckdb.md`
  - `docs/integrations/local-bi.md`
  - `docs/accountant-handoff.md`

### Retention and privacy controls

- `LocalAppSettings.rawStatementRetentionDays` defaults to `90`.
- Setting the value to `0` keeps raw statement payload rows indefinitely.
- Positive values are used to prune `raw_statement_items` only; normalized
  ledger rows are never deleted by raw-payload retention.
- New privacy/compliance notes document local-use boundaries, Ukrainian
  personal data assumptions, and support-bundle retention:
  - `docs/privacy-retention-review.md`
  - `docs/compliance/local-use-privacy-review.md`
  - `docs/compliance/ukrainian-personal-data-review.md`
  - `docs/support-bundle-retention.md`

## Provider/FOP Spike

Provider mode remains a spike in this release. It is useful for validating the
technical boundary, not for replacing the simple personal-token flow.

- `src/monobank/provider.ts` contains isolated secp256k1 signing helpers.
- Golden-vector tests cover payload generation, signing, public-key derivation,
  signed headers, and signature verification.
- The mock Monobank server covers provider registration, registration status,
  company settings, access request/status, client-info, and statement calls.
- The Settings prototype is hidden behind `?provider_spike=1` or
  `localStorage.mono-ledger-provider-spike=1`.
- Provider private-key field names, `X-Sign`, and `X-Key-Id` are redacted from
  logs and privacy helpers.

Provider mode does not graduate because safe persistent provider-key storage and
clear product demand are not proven yet. The personal Monobank token path stays
the default and supported path.

## Developer Notes

- Runtime dependencies added:
  - `hyparquet-writer` for Parquet generation.
  - `@noble/secp256k1` for the provider signing boundary.
- Dev dependency added:
  - `hyparquet` for Parquet metadata and row-count verification in tests.
- `ExportFormat` now includes `parquet` and `sqlite`.
- `LedgerExport.body` can be a `string` or `Uint8Array`.
- `createSqliteSnapshotExport` returns a redacted SQLite database payload for
  local downloads or configured folder saves.
- The package version and public `version` export are now `0.4.0`.

## Upgrade Notes

This is a minor version bump from `0.3.0` to `0.4.0`. Existing local databases
will apply migrations `0025_raw_statement_retention` and `0026_bi_views` on
startup. No manual action is required.

Users who share exports should prefer Parquet for BI workflows and redacted
SQLite snapshots for broad accountant review. Full raw payload archives remain
available through the existing JSONL export path.
